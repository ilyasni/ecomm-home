# ADR-0001: Data Ownership между Medusa и Strapi

## Context

Проект использует `Medusa` как commerce backend и `Strapi` как headless CMS.  
Ранее в интеграционном документе часть коммерческих сущностей (например, `Order`, `Address`, операции по бонусам) описывалась в контуре Strapi как первичные записи. Это создаёт риск двойного владения данными, рассинхрона цен/остатков и усложняет сопровождение.

Best practices из Context7 (Medusa + Strapi) указывают на необходимость чёткой границы владения:  
- commerce-сущности — в Medusa;  
- editorial/content-сущности — в Strapi;  
- синхронизация между системами — событийная, идемпотентная, с retry/reconciliation.

## Decision

Принять модель **single source of truth**:

1. **Medusa — master для коммерческих данных**
   - товары, варианты, цены, остатки;
   - корзины, checkout, заказы, платежи, доставки, возвраты;
   - покупатели магазина и адреса для заказов.

2. **Strapi — master для контентных данных**
   - контентные страницы, статьи, баннеры, SEO;
   - editorial-блоки карточек/категорий;
   - контентные медиа.

3. **Связь сущностей**
   - в Strapi использовать ссылки вида `medusa_product_id` / `medusa_collection_id`;
   - Strapi не хранит цены/остатки/order lifecycle как первичные поля.

4. **Топология БД**
   - отдельная БД для Medusa;
   - отдельная БД для Strapi;
   - отдельные роли доступа, секреты и backup-policy.

## Consequences

### Плюсы

- нет двойного владения критичными коммерческими сущностями;
- ниже риск рассинхрона цен/остатков/заказов;
- проще аудит и rollback инцидентов;
- предсказуемая эксплуатация и масштабирование.

### Минусы / Trade-offs

- появляется отдельный sync-слой между Medusa и Strapi;
- требуется поддержка контрактов событий и мониторинг синка;
- витрина агрегирует данные из двух источников.

## Integration Contract

1. События Medusa (`product.*`, `variant.*`, `price.*`, `inventory.*`) обрабатываются sync-worker.
2. Sync-worker выполняет upsert в Strapi только для проекций и контента, связанных по `medusa_*_id`.
3. Все операции sync — идемпотентные, с retry и DLQ.
4. Ночной reconciliation job проверяет консистентность ключевых связей.

## Guardrails

- Запрещено создавать в Strapi мастер-сущности заказов и клиентов магазина.
- Запрещено редактировать цену/остаток в Strapi как источник истины.
- При конфликте данных приоритет у Medusa для commerce-полей.

## Rollback

1. Отключить sync-worker feature flag.
2. Переключить storefront на Medusa-only данные для коммерческих блоков.
3. Восстановить Strapi content projection из последнего валидного snapshot.

## Status

Accepted
